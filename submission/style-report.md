# Что пришлось добавить, чтобы ИИ не съезжал со стиля

## 1. Явная языковая политика

Было зафиксировано, что описания должны быть:

- на русском языке;
- в официальном нейтральном тоне;
- без разговорных оборотов;
- без маркетинговых или эмоциональных формулировок.

Без этой фиксации модель легко уходит в смешанный стиль: часть текста на английском, часть на русском, а примеры и пояснения становятся разговорными.

## 2. Жесткая структура для каждого описания

Для удержания единообразия была задана повторяемая схема:

1. Назначение.
2. Основные сущности.
3. Публичные методы.
4. Параметры методов.
5. Примеры использования.
6. Практические замечания.

Это убирает типичную проблему генеративного дрейфа, когда разные сущности описываются в разных форматах.

## 3. Привязка к реальному коду, а не к домыслам

Чтобы модель не придумывала поведение, в prompt и в workflow было важно опираться на:

- `src/query_engine.py` как исходный объект документирования;
- `src/main.py` как источник реальных CLI-сценариев;
- `tests/test_porting_workspace.py` как источник проверяемых примеров использования.

Именно эта привязка позволяет получить документацию, которая совпадает с фактическим поведением модуля.

## 4. Разделение ролей между skill и tool

Для стабильного результата была выбрана двухслойная схема:

- skill отвечает за стиль, тон и структуру текста;
- MCP tool отвечает за анализ модуля и валидацию документа.

Если оставить только prompt/skill, модель может выдерживать стиль, но чаще ошибается в полноте покрытия. Если оставить только tool, результат становится механическим и теряет редакционное качество. Комбинация дает более устойчивый результат.

## 5. Как происходило взаимодействие

Взаимодействие строилось по следующей схеме:

1. Сначала был выбран репозиторий и подтверждена совместимость с Qwen CLI.
2. Затем был зафиксирован план и уточнены deliverables.
3. После этого были реализованы:
   - документация модуля;
   - skill и команда для Qwen;
   - MCP tool server для анализа и проверки результата.

Итоговый подход можно повторно использовать для документирования других Python-модулей, если сохранить ту же структуру и валидацию.
